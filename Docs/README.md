## 📝 文档编写规范

> 💡 **提示**: 以下为文档编写规范，供编写文档时参考。日常查找文档请使用上方索引表。

### 📋 文档分类

项目文档根据内容性质分为以下六类：

### 1. 分类索引文档（README.md）

**用途**: 用于目录组织和文档导航

**位置**: 各系统/模块文件夹根目录

**结构规范**:
- 📖 **文档列表** - 列出核心文档（总览、关键技术设计、使用指南），每项1-2句简介
- 🚀 **开发进展** - 链接到进展文档，标注版本和状态
- 🔗 **相关文档** - 跨系统链接
- 📊 **系统状态** - 表格展示核心模块完成度

**格式要求**:
- 使用 emoji 区分章节
- 文档简介控制在1-2句话
- 核心内容用要点列表
- 提供返回链接到文档中心

---

### 2. 系统总览文档

**用途**: 系统整体设计、架构概述、核心概念说明

**命名规范**: `[系统名称]-Overview [系统总览].md` 或 `[系统名称]-System [系统名称].md`

**结构规范**:
- 开头：版本、适用范围、面向读者（引用格式）
- **TL;DR**：5-8行要点
- 概述：1-2段说明核心目标和价值
- 核心概念：定义和作用
- 架构设计：架构图、数据流
- 系统关系：与其他系统的集成方式
- 末尾：元信息（版本、时间、状态、Owner、变更摘要）

**格式要求**:
- 使用编号标题组织大章节
- 代码块展示架构图、数据流
- **精简原则**：概述避免冗余铺垫，直接说明核心目标

---

### 3. 技术设计文档

**用途**: 具体技术实现方案、数据结构、核心类设计

**命名规范**: `[系统名称]-[功能名称]-Design [设计说明].md`

**结构规范**:
- 概述：1-2段简要说明
- 架构设计：组件/类职责、数据结构、设计要点
- 流程/实现细节：流程图/伪代码
- 典型示例（可选）：可运行代码
- 关键决策（轻量ADR）：问题/备选/选择/影响

**格式要求**:
- 聚焦核心逻辑，使用**加粗**标注关键概念
- 代码块展示流程图
- **精简原则**：概述1-2段，避免装饰性文字
- **禁止内容**：
  - ❌ 不要包含"未来展望"、"扩展方向"等模块
  - ❌ **不要包含进展性内容**：当前版本、开发进展、完成状态、Phase划分、预计工时、TODO清单等应放在专门的进展文档中
- **追溯要求**：链接上游需求/任务与下游实现/PR/代码文件

---

### 4. 使用指南文档

**用途**: 工具使用方法、操作步骤、快速开始指南

**命名规范**: `[工具名称]-Guide [使用指南].md` 或 `[工具名称]-Usage [使用说明].md`

**结构规范**:
- 开头：工具路径、依赖库、状态（引用格式）
- 概述：工具简介、核心特性
- 快速开始：编号步骤，每步包含说明和可运行代码示例
- 详细功能：功能说明和使用示例
- 注意事项：重要警告

**格式要求**:
- 使用编号步骤组织流程
- 代码示例必须可运行
- **精简原则**：删除重复说明，聚焦核心步骤

---

### 5. 配置规范文档

**用途**: 配置表结构、字段定义、格式规范

**命名规范**: `[配置名称]-Config [配置说明].md` 或 `Table-Config [表格配置说明].md`

**结构规范**:
- 概述：配置用途和重要性
- 架构：设计理念、配置分类（树状结构）
- 详细说明：配置表用途、关键字段、关联关系
- 字段定义：表格展示字段名、类型、说明、默认值
- 格式规范：基本格式要求、数据类型规范（表格）
- 注意事项：禁止操作、格式要求

**格式要求**:
- 使用 emoji 区分章节
- 表格展示字段定义和类型规范
- **精简原则**：只列出必要配置说明，删除示例性配置和重复描述

---

### 6. 开发进展文档

**用途**: 开发状态跟踪、版本记录、完成度评估

**命名规范**: `[系统名称]-Progress [开发进展].md`

**结构规范**:
- 开头：当前版本、最后更新、负责人
- TL;DR（四象限）：状态/进度、已达成、风险/阻塞、下一步
- 版本历史：版本号、状态、完成内容（复选框）、待完成、预计工时
- 当前阶段：阶段名称、完成度、下一步计划

**格式要求**:
- 使用复选框跟踪任务
- **精简原则**：只记录关键信息，聚焦实际进展
- **追溯要求**：回链到技术文档与代码变更/PR

## 📝 内容编写原则

### 核心原则：精简、聚焦、实用

文档编写的首要目标是**聚焦核心逻辑，减少不必要的废话**。

#### 1. 精简原则

- ✅ **直接说明核心内容** - 开门见山，避免冗余铺垫
- ✅ **删除重复说明** - 每个要点只说一次
- ✅ **避免过度解释** - 假设读者具备基本技术背景
- ❌ **避免**："本文档将详细介绍..."、"接下来我们将..."等废话
- ❌ **避免**：重复强调已经说明过的内容

#### 2. 聚焦原则

- ✅ **突出核心逻辑** - 重点说明"是什么"和"为什么"
- ✅ **结构清晰** - 使用层级标题明确组织
- ✅ **关键信息前置** - 最重要的内容放在最前面
- ❌ **避免**：大段背景介绍和无关内容
- ❌ **避免**：过多的装饰性文字

#### 3. 实用原则

- ✅ **可执行性** - 提供可直接使用的代码示例和配置
- ✅ **可查找性** - 关键信息易于快速定位
- ✅ **可维护性** - 结构清晰，便于后续更新
- ❌ **避免**：过多的理论阐述和概念辨析
- ❌ **避免**：无法直接使用的占位代码
- ❌ **禁止**：**不要包含"未来展望"、"扩展方向"、"后续规划"等模块** - 这些内容没有实际作用，浪费token且可能误导开发方向

#### 内容检查标准

编写完成后，自问：
1. 每个段落是否都包含**核心信息**？
2. 是否有**重复**或**冗余**的说明？
3. 读者能否**快速找到**需要的信息？
4. 是否删除了所有**装饰性废话**？
5. **是否删除了"未来展望"、"扩展方向"等无用模块**？

## 🎨 通用格式规范

### Markdown 格式要求

1. **标题层级**: 
   - 一级标题（#）用于文档标题
   - 二级标题（##）用于主要章节
   - 三级标题（###）用于子章节

2. **代码块**:
   - 使用语言标识：\`\`\`语言
   - 架构图使用文本图或代码块
   - 确保代码示例可运行

3. **列表**:
   - 无序列表用于要点说明
   - 有序列表用于步骤说明
   - 使用 `- [ ]` 或 `- [x]` 表示任务状态

4. **表格**:
   - 使用表格展示结构化数据
   - 对齐表格列
   - 表头清晰

5. **强调**:
   - **加粗**用于关键概念
   - *斜体*用于补充说明
   - `代码`用于技术术语

### Emoji 使用规范

| Emoji | 用途 |
|-------|------|
| 📖 | 文档/说明 |
| 📋 | 列表/清单 |
| 🚀 | 开发进展 |
| 🔗 | 相关链接 |
| 📊 | 状态/数据 |
| ⚠️ | 警告/注意事项 |
| ✅ | 完成/正确 |
| ❌ | 错误/禁止 |
| 🔧 | 工具/配置 |
| 📅 | 日期 |
| 👥 | 团队/读者 |
| 🎯 | 目标/适用范围 |

### 图表与流程表达
- 统一使用简洁 ASCII 或 Mermaid（二选一），避免混用
- 优先用图表达流程/状态/时序，正文仅保留关键约束

### 追溯要求
- 文档需包含跨文档链接：上游需求/任务单 → 本文档 → 代码实现/PR
- 进展文档需回链对应技术文档与代码变更

### 文档元信息

每篇文档应在末尾包含：

```markdown
---

*文档版本：v1.0*  
*创建时间：YYYY-MM-DD*  
*最后更新：YYYY-MM-DD*  
*状态：[策划案/实现中/已完成]*
```

**重要要求**:
- **文档更新时间应当准确** - 每次修改文档时，必须将"最后更新"日期更新为实际修改日期（格式：YYYY-MM-DD）
- 创建时间和最后更新日期应当使用实际日期，不要使用占位符
- 只有在首次创建时才填写"创建时间"，后续只更新"最后更新"日期
- 所有示例均需为最小可运行，且指向项目内真实文件/路径/类型
- 文档需标注 *Owner* 与一次性**变更摘要**（一句话）

### 文件命名规范

- 使用中文描述性名称
- 英文别名使用 `-` 分隔单词
- 格式：`[主标题] [英文别名].md`
- 示例：`AI-StateMachine-Design AI状态机设计.md`
