# Astrum 文档编写规范

> 📚 本文档规定了 Astrum 项目文档的编写规范和格式要求

## 📋 文档分类

项目文档根据内容性质分为以下六类：

### 1. 分类索引文档（README.md）

**用途**: 用于目录组织和文档导航

**位置**: 各系统/模块文件夹根目录

**结构规范**:
```markdown
# [系统名称]文档

## 📖 文档列表

### [文档标题](文档路径.md)
文档简介

**核心内容**:
- 要点1
- 要点2
- 要点3

---

## 🚀 开发进展
- [进展文档](路径.md) - ✅ **版本号** (状态说明)

---

## 🔗 相关文档
- [相关系统](../路径/) - 关系说明

---

## 📊 系统状态
| 模块 | 状态 | 完成度 |
|------|------|--------|
| 模块1 | ✅ 完成 | 90% |

---

**返回**: [文档中心](../README.md)
```

**格式要求**:
- 使用 emoji 图标区分不同章节（📖 🚀 🔗 📊 等）
- 使用表格展示模块状态
- 提供返回链接
- **精简原则**：文档简介控制在1-2句话，核心内容用要点列表，避免冗长描述

---

- 末尾包含元信息（版本、时间、状态）
- **精简原则**：概述部分避免冗余铺垫，直接说明核心目标和价值

## 📝 内容编写原则

### 核心原则：精简、聚焦、实用

文档编写的首要目标是**聚焦核心逻辑，减少不必要的废话**。

#### 1. 精简原则

- ✅ **直接说明核心内容** - 开门见山，避免冗余铺垫
- ✅ **删除重复说明** - 每个要点只说一次
- ✅ **避免过度解释** - 假设读者具备基本技术背景
- ❌ **避免**："本文档将详细介绍..."、"接下来我们将..."等废话
- ❌ **避免**：重复强调已经说明过的内容

#### 2. 聚焦原则

- ✅ **突出核心逻辑** - 重点说明"是什么"和"为什么"
- ✅ **结构清晰** - 使用层级标题明确组织
- ✅ **关键信息前置** - 最重要的内容放在最前面
- ❌ **避免**：大段背景介绍和无关内容
- ❌ **避免**：过多的装饰性文字

#### 3. 实用原则

- ✅ **可执行性** - 提供可直接使用的代码示例和配置
- ✅ **可查找性** - 关键信息易于快速定位
- ✅ **可维护性** - 结构清晰，便于后续更新
- ❌ **避免**：过多的理论阐述和概念辨析
- ❌ **避免**：无法直接使用的占位代码
- ❌ **禁止**：**不要包含"未来展望"、"扩展方向"、"后续规划"等模块** - 这些内容没有实际作用，浪费token且可能误导开发方向

#### 内容检查标准

编写完成后，自问：
1. 每个段落是否都包含**核心信息**？
2. 是否有**重复**或**冗余**的说明？
3. 读者能否**快速找到**需要的信息？
4. 是否删除了所有**装饰性废话**？
5. **是否删除了"未来展望"、"扩展方向"等无用模块**？

## 🎨 通用格式规范

### Markdown 格式要求

1. **标题层级**: 
   - 一级标题（#）用于文档标题
   - 二级标题（##）用于主要章节
   - 三级标题（###）用于子章节

2. **代码块**:
   - 使用语言标识：\`\`\`语言
   - 架构图使用文本图或代码块
   - 确保代码示例可运行

3. **列表**:
   - 无序列表用于要点说明
   - 有序列表用于步骤说明
   - 使用 `- [ ]` 或 `- [x]` 表示任务状态

4. **表格**:
   - 使用表格展示结构化数据
   - 对齐表格列
   - 表头清晰

5. **强调**:
   - **加粗**用于关键概念
   - *斜体*用于补充说明
   - `代码`用于技术术语

### Emoji 使用规范

| Emoji | 用途 |
|-------|------|
| 📖 | 文档/说明 |
| 📋 | 列表/清单 |
| 🚀 | 开发进展 |
| 🔗 | 相关链接 |
| 📊 | 状态/数据 |
| ⚠️ | 警告/注意事项 |
| ✅ | 完成/正确 |
| ❌ | 错误/禁止 |
| 🔧 | 工具/配置 |
| 📅 | 日期 |
| 👥 | 团队/读者 |
| 🎯 | 目标/适用范围 |

### 文档元信息

每篇文档应在末尾包含：

```markdown
---

*文档版本：v1.0*  
*创建时间：YYYY-MM-DD*  
*最后更新：YYYY-MM-DD*  
*状态：[策划案/实现中/已完成]*
```

**重要要求**:
- **文档更新时间应当准确** - 每次修改文档时，必须将"最后更新"日期更新为实际修改日期（格式：YYYY-MM-DD）
- 创建时间和最后更新日期应当使用实际日期，不要使用占位符
- 只有在首次创建时才填写"创建时间"，后续只更新"最后更新"日期

### 文件命名规范

- 使用中文描述性名称
- 英文别名使用 `-` 分隔单词
- 格式：`[主标题] [英文别名].md`
- 示例：`AI-StateMachine-Design AI状态机设计.md`

在提交文档前，请检查：

- [ ] 标题层级清晰，符合规范
- [ ] 代码示例可运行或已标注为伪代码
- [ ] 表格格式正确，对齐美观
- [ ] 链接路径正确，可正常跳转
- [ ] 包含了文档元信息（版本、日期、状态）
- [ ] **文档更新时间准确**（修改后必须更新"最后更新"日期为实际日期）
- [ ] **内容精简聚焦**（删除冗余废话，聚焦核心逻辑）
- [ ] **禁止未来展望模块**（技术文档不要包含"扩展方向"、"未来展望"等内容）
- [ ] 使用合适的 emoji 图标
- [ ] 没有错别字和格式错误
